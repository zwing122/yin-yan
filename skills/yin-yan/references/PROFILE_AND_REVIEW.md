# 用户画像和审查

## 用户画像可沉淀内容

用户画像只沉淀能稳定改善记账体验的偏好和约束。

可沉淀：

- 常用账户或支付方式，但必须来自用户明确表达。
- 固定收入、房租、订阅、还款日等周期性事项。
- 预算偏好，例如“咖啡每天最多 2 笔”“餐饮每月 800”。
- 分类偏好，例如用户确认“便利店饮料都算餐饮 / 饮料”。
- 低打扰偏好，例如“不要每笔都问支付方式”。

不可沉淀：

- 用户没有说出的支付方式、账户、地点、具体日期。
- 商户或品牌作为独立画像维度。
- 从单次消费推断长期习惯。
- `冲动消费`、`乱花钱`、`不理性`等人格化或道德化标签。
- 敏感身份、健康、关系、职业细节，除非用户明确要求用于记账。

## 低打扰追问

默认原则：能安全记录就先记录，不为低风险缺口打断用户。

必须追问或标记审查：

- 金额缺失。
- 收入/支出方向不清。
- 可能重复。
- 到期事项没有日期。
- 金额异常或明显超出历史范围。
- 分类会影响预算判断且无法保守归类。

不追问，直接留空：

- 支付方式未填写。
- 账户未填写。
- 地点未填写。
- 用户没有说出的具体日期。

示例：

```text
午饭花了25 -> 已记录：正餐 ¥25。支付方式未填写
瑞幸18 -> 已记录：咖啡 ¥18。支付方式未填写
下周还信用卡 -> 需要日期或进入待审查
```

## 审查优先级

从高到低：

1. 可能重复：同金额、同类目、近时间输入。
2. 到期风险：未来付款、还款、订阅、账单。
3. 预算超限：金额预算或每日笔数预算。
4. 金额异常：明显高于近期同类支出。
5. 分类不确定：影响统计或预算。
6. 普通缺口：账户、支付方式、地点等。

## 画像更新口径

画像更新必须能回答两个问题：

- 这个信息是否来自用户明确表达或反复确认？
- 写入后是否会减少未来打扰或提升统计准确性？

如果答案不明确，不写入画像，只保留在单笔事件或审查建议中。

## 重点提醒口径

重点提醒只在汇总、现金流心跳和定时提醒中生成，不作为用户画像持久化。网页端负责更完整的展示、解释和账本校验。

第一版允许提示：

- 高频低值：小额支出多次出现。
- 预算消耗过快：预算进度过快或每日笔数触顶。
- 订阅/固定支出累积。
- 延迟付款集中。
- 大额低提及：只针对用户确认追踪的大额物品。

IM 文案要保留“仅基于已记录数据”的边界，给出可执行的小动作，不评价用户，不使用抽象总称。
